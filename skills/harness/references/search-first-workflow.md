# Search-First 코딩 워크플로우

> 하네스에서 **코드 작성을 수행하는 에이전트와 스킬**에 적용하는 5단계 패턴.
> 맹목적 코드 작성(Code-First 안티패턴)을 방지하고 최적 경로를 선택한다.

**핵심 원칙:** 존재하는 것을 다시 만들지 않는다. 먼저 찾고, 평가하고, 그 다음 만든다.

## 1. 패턴 개요

```
요청 수신
  ↓
Step 1: 니즈 분석 — 요구사항 분해
  ↓
Step 2: 병렬 검색 — 내부 코드 / 표준 라이브러리 / 외부 패키지·MCP 동시 탐색
  ↓
Step 3: 평가 — 검색 결과의 적합성·비용·통합 난이도 판단
  ↓
Step 4: 결정 — 직접 구현 vs 외부 패키지 도입
  ↓
Step 5: 구현 — 결정된 최적 경로로 코드 작성
```

> **안티패턴:** 요청 → 즉시 코드 작성 (Code-First)

---

## 2. Step 1 — 니즈 분석

구현 요청을 받으면 코드를 쓰기 전에 요구사항을 분해한다.

| 분석 항목 | 확인 내용 |
|----------|----------|
| **핵심 기능** | 무엇을 해야 하는가 |
| **제약 조건** | 언어·프레임워크·라이선스·성능·번들 크기 제한 |
| **통합 포인트** | 기존 코드베이스와 어떻게 연결되는가 |
| **범위 경계** | MVP vs 이상적 구현 구분 |

---

## 3. Step 2 — 병렬 검색

요구사항이 명확해지면 **동시에** 3개 영역을 탐색한다.

### 2-A. 프로젝트 내부 탐색

이미 같은 기능을 하는 코드, 유사한 패턴, 공통 유틸리티 함수가 있는가?

```
Glob("**/*.{ext}")
Grep("유사 함수명 / 클래스명")
SemanticSearch("기능 설명")
```

### 2-B. 표준 라이브러리 / 빌트인 확인

- 언어 표준 라이브러리에서 제공하는가?
- 현재 사용 중인 프레임워크가 이미 포함하는가?
- 플랫폼 내장 API로 해결 가능한가?

### 2-C. 외부 패키지·MCP 스캔

- npm / pip / cargo / maven 등 패키지 레지스트리
- GitHub 인기 오픈소스 (별점, 최근 업데이트, 이슈 활성도)
- MCP(Model Context Protocol) 서버 — 이미 래핑된 통합이 있는가?

---

## 4. Step 3 — 평가

찾은 후보를 다음 기준으로 채점한다.

| 기준 | 확인 항목 |
|------|----------|
| **기능 적합성** | 요구사항의 몇 %를 충족하는가? |
| **유지보수 상태** | 최근 커밋·이슈 대응이 활발한가? |
| **번들 비용** | 크기·의존성 트리가 감당 가능한가? |
| **라이선스** | 프로젝트 정책과 충돌하지 않는가? |
| **통합 난이도** | 기존 스택과 얼마나 자연스럽게 연결되는가? |

---

## 5. Step 4 — 결정

평가 결과에 따라 명확히 결정하고 **이유를 사용자에게 설명**한다.

### 외부 패키지 도입 (권장 조건)
- 요구사항 80% 이상 충족하는 검증된 패키지가 존재할 때
- 직접 구현보다 유지보수·보안 업데이트 부담이 낮을 때
- 번들 비용이 허용 범위 내일 때

### 직접 구현 (권장 조건)
- 요구사항에 맞는 패키지가 없거나 과도하게 무거울 때
- 프로젝트 내부 패턴과의 일관성이 중요할 때
- 의존성 추가 자체가 리스크일 때 (보안 정책, 번들 크기 제한)

### 결정 보고 형식

에이전트·스킬은 결정 결과를 다음 형식으로 출력한다:

```markdown
## 검색 결과
- 내부: {발견한 유사 코드 또는 "없음"}
- 표준 라이브러리: {해당 항목 또는 "없음"}
- 외부 패키지 후보: {패키지명 — 적합성 점수/100}

## 결정
{외부 패키지 도입 / 직접 구현}

## 이유
{1~2문장}
```

---

## 6. Step 5 — 구현

Step 4의 결정에 따라 구현한다.

### 외부 패키지 도입 시
1. 설치 명령 제시
2. 최소 통합 코드 작성 (래퍼 또는 어댑터)
3. 기존 코드베이스 패턴과 일치하도록 스타일 맞춤

### 직접 구현 시
1. Step 2-A에서 발견한 내부 패턴 재사용·확장
2. 새 코드는 기존 컨벤션 준수
3. 향후 패키지 대체가 쉽도록 인터페이스 분리

### 구현 후 체크
- [ ] 동일 기능 중복 구현 없음
- [ ] 불필요한 의존성 추가 없음
- [ ] 기존 코드베이스 컨벤션 준수

## 7. 에이전트·스킬 작성 시 통합 방법

하네스에서 코딩 에이전트와 작업 스킬을 생성할 때 Search-First를 포함하는 방법.

### 에이전트 정의 파일에 통합

코드 작성 역할의 에이전트 `.md` / `.toml` 파일에 다음 섹션을 추가한다:

```markdown
## 코딩 원칙 — Search-First

코드를 작성하기 전에 반드시 다음 순서를 따른다:

1. **니즈 분석** — 요구사항·제약·통합 포인트 명확화
2. **병렬 검색** — 내부 코드(Glob/Grep/SemanticSearch) + 표준 라이브러리 + 외부 패키지·MCP 동시 탐색
3. **평가** — 기능 적합성·유지보수·번들 비용·라이선스 채점
4. **결정** — 직접 구현 vs 외부 패키지 도입, 이유 명시
5. **구현** — 결정된 경로로만 코드 작성

> 안티패턴: 요청 즉시 코드 작성 금지. 반드시 Step 1~4 완료 후 Step 5 진입.
```

### 작업 스킬 본문에 통합

코드를 생성·수정하는 work skill의 **첫 번째 Step**으로 삽입한다:

```markdown
## Step 1: Search-First 사전 조사

코드 작성 전 다음 탐색을 완료한다:

- `Glob / Grep / SemanticSearch` — 프로젝트 내 유사 구현 확인
- 프레임워크 내장 기능 확인
- 외부 패키지 후보 3개 이내로 평가

결과를 `artifacts/search-result.md`에 기록 후 다음 Step 진행.
```

### 오케스트레이터 연결

오케스트레이터의 **Phase 1 (준비)** 에 Search-First 결과를 task-board에 포함시킨다:

```markdown
### Phase 1: 준비
1. 입력 분석
2. **Search-First 사전 조사** (`artifacts/search-result.md` 생성)
   - 내부 코드·표준 라이브러리·외부 패키지 병렬 탐색
   - 직접 구현 / 외부 패키지 결정 및 근거 기록
3. task-board.md 작성 (search-result.md 기반으로 구현 범위 확정)
```

## 8. 안티패턴

| 안티패턴 | 올바른 대응 |
|----------|-----------|
| "간단하니까 그냥 만들자" | 표준 라이브러리 확인 먼저 |

| "패키지 쓰면 무거워지니까" | 번들 비용 실제 측정 후 결정 |
| "구현하면서 찾아보면 되지" | Step 2 완료 후 Step 5 진입 |
