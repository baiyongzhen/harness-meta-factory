## 1. 실행 후 피드백 수집

매 하네스 실행 완료 후 피드백 기회를 제공한다.

- "결과 개선할 부분이 있나요?"
- "워크플로우 바꾸고 싶은 점이 있나요?"

피드백이 없으면 넘어간다. **강요하지 않되, 반드시 기회는 제공한다.**

---

## 2. 피드백 반영 경로

피드백 유형에 따라 수정 대상이 다르다:

| 피드백 유형 | 수정 대상 | 예시 |
|-----------|----------|------|
| 결과물 품질 | 해당 에이전트의 스킬 | "분석이 너무 피상적" → 스킬에 깊이 기준 추가 |
| 에이전트 역할 | 에이전트 정의 `.md` | "보안 검토도 필요" → 새 에이전트 추가 |
| 워크플로우 순서 | 오케스트레이터 스킬 | "검증을 먼저 해야" → Phase 순서 변경 |
| 팀 구성 | 오케스트레이터 + 에이전트 | "이 둘은 합쳐도 될 듯" → 에이전트 병합 |
| 트리거 누락 | 스킬 description | "이 표현으로 하면 작동 안 함" → description 확장 |

---

## 3. 변경 이력 관리

모든 변경은 **entry file 변경 이력**에 기록한다 (CLAUDE.md / AGENTS.md / GEMINI.md).

```markdown
**변경 이력:**
| 날짜 | 변경 내용 | 대상 | 사유 |
|------|----------|------|------|
| 2026-04-05 | 초기 구성 | 전체 | - |
| 2026-04-07 | QA 에이전트 추가 | agents/qa.md | 산출물 품질 검증 부족 피드백 |
| 2026-04-10 | 톤 가이드 추가 | skills/content-creator | "너무 딱딱하다" 피드백 |
```

이 이력을 통해 하네스가 어떤 방향으로 진화했는지 추적하고, 퇴행(regression)을 방지한다.

---

## 4. 진화 트리거

사용자가 명시적으로 "하네스 수정해줘"라고 할 때만이 아니라, 다음 상황에서도 진화를 제안한다:

- 같은 유형의 피드백이 **2회 이상 반복**될 때
- 에이전트가 **반복적으로 실패**하는 패턴이 발견될 때
- 사용자가 **오케스트레이터를 우회**하여 수동으로 작업하는 것이 관찰될 때

---

## 5. 운영·유지보수 워크플로우

기존 하네스의 점검·수정·동기화를 체계적으로 수행한다.
Phase 0에서 "운영/유지보수" 분기로 진입했을 때 이 워크플로우를 따른다.

**Step 1: 현황 감사**
- 대상 플랫폼 agents/, skills/, entry file 읽기
- entry file ↔ 실제 파일 drift 감지
- 현재 문제점 목록 작성

**Step 2: 점진적 추가·수정**
- Phase 선택 매트릭스 적용 (SKILL.md Phase 0)
- 영향받는 에이전트·스킬만 수정, 나머지는 유지

**Step 3: Entry file 변경 이력 갱신**
- 수행한 모든 변경을 날짜·내용·대상·사유로 기록

**Step 4: 변경 검증**
- SKILL.md Phase 6 검증 기준 적용
- 수정된 에이전트·스킬에 대한 트리거 검증 재실행
