## 1. 왜 토큰 최적화인가

에이전트 팀 하네스는 **복수의 독립 인스턴스**가 동시에 실행된다. 각 인스턴스가 컨텍스트를 공유하거나 대용량 파일을 읽을 때 비용이 기하급수적으로 증가한다.

```
단일 에이전트   : 토큰 × 1
3인 팀 (fork)   : 토큰 × 3   ← 컨텍스트 공유 시
3인 팀 (empty)  : 토큰 × 에이전트별 작업 컨텍스트
```

**비용 폭발 시나리오:** 500줄 SKILL.md × 5개 스킬 × 3명 팀원 = 7,500줄이 컨텍스트에 동시 적재.

## 2. 하네스에서 토큰 비용이 발생하는 곳

| 비용 발생 위치 | 원인 | 위험도 |
|--------------|------|--------|
| SKILL.md 본문 | 스킬 트리거 시 전체 로드 | ★★★ |
| 에이전트 컨텍스트 전달 (`context: fork`) | 이전 대화 전체 복사 | ★★★ |
| references/ 파일 불필요 로드 | 조건 없이 전부 Read | ★★★ |
| Handoff 파일 누적 | `_workspace/`·`artifacts/` 비대화 | ★★ |
| 에이전트 정의 파일 과대 | 수백 줄 agent.md | ★★ |
| 팀 전체 브로드캐스트 | `SendMessage({to: "all"})` 남용 | ★★ |
| 아카이브 미실행 | 이전 실행 파일 잔류 | ★ |

## 3. 레이어별 최적화 전략

### 3-1. SKILL.md 크기 관리

**목표: 본문 300줄 이하 (한계: 500줄)**

```
스킬 SKILL.md  ← 핵심 워크플로우만 (결정 트리, 핵심 규칙)
└── references/
    ├── detail-a.md   ← "A 시나리오일 때만 이 파일을 읽어라"
    └── detail-b.md   ← "B 시나리오일 때만 이 파일을 읽어라"
```

**포인터 패턴 (조건부 로딩):**
```markdown
> AWS 배포 시: `references/aws.md` 읽기
> GCP 배포 시: `references/gcp.md` 읽기
> Azure 배포 시: `references/azure.md` 읽기
```
→ 3개 파일 중 실제로 읽히는 건 1개. **67% 절약**.

### 3-2. 에이전트 컨텍스트 선택

에이전트를 호출할 때 `context` 옵션을 신중히 선택한다:

| 옵션 | 토큰 비용 | 언제 사용 |
|------|----------|----------|
| `context: fork` | 높음 — 현재 컨텍스트 전체 복사 | 이전 분석 결과가 필수적일 때 |
| `context: empty` | 낮음 — 빈 컨텍스트에서 시작 | 독립 태스크, 파일 handoff로 충분할 때 |

**기본값: `context: empty` + 파일 handoff**
이전 에이전트의 결과를 `_workspace/` 또는 `artifacts/`에 저장하고,
다음 에이전트 프롬프트에 경로를 전달하는 것이 대부분의 경우 충분하다.

```markdown
# 오케스트레이터 지시 예시
- 분석 결과를 `_workspace/analysis.md`에 저장
- QA 에이전트 호출: context: empty, prompt에 `_workspace/analysis.md` 경로 포함
```

### 3-3. 에이전트 정의 파일 최적화

에이전트 `.md` / `.toml` 파일은 **100줄 이하**를 목표로 한다.

**포함할 것:**
- 역할 (2~3문장)
- 핵심 원칙 (5개 이하 bullet)
- 입출력 경로 (표)
- 에러 핸들링 (간략)

**제거할 것:**
- 상세 방법론 → work skill로 이동
- 예시 코드 → skill references/로 이동
- 플랫폼 공통 규칙 → 오케스트레이터 또는 skill로 이동

### 3-4. Handoff 파일 크기 관리

`_workspace/` · `artifacts/` 파일이 비대해지면 에이전트가 Read할 때마다 토큰 비용이 누적된다.

**크기 가이드라인:**

| 파일 | 권장 크기 | 초과 시 조치 |
|------|----------|------------|
| `00-input.md` | ~50줄 | 핵심 요구사항만, 상세 데이터는 별도 파일 |
| `task-board.md` | ~30줄 | 완료 항목 주기적 축소 |
| `02_{role}.md` | ~200줄 | 초과 시 요약본과 상세본 분리 |
| `final-report.md` | ~300줄 | 초과 시 섹션별 분리 파일 + 인덱스 |

**요약본 패턴:**
```
artifacts/
├── 02_research-summary.md    ← 에이전트 간 공유용 (50줄 이하)
└── 02_research-full.md       ← 상세 보관용 (제한 없음)
```
에이전트들은 summary를 읽고, full은 특정 정보 필요 시에만 참조한다.

### 3-5. references/ 조건부 로딩 강제

references/ 파일을 무조건 전부 읽지 않는다. SKILL.md 본문에 **명확한 로딩 조건**을 명시한다:

```markdown
## 플랫폼별 상세

- AWS 환경: `references/aws.md` 읽기
- 멀티 리전: `references/multi-region.md` 읽기
- 온프레미스: `references/on-premise.md` 읽기

> 해당하지 않는 파일은 읽지 않는다.
```

## 4. 모델 선택 최적화

에이전트별 작업 복잡도에 맞는 모델을 선택하면 **품질 유지 + 비용 절감**을 동시에 달성한다.

| 작업 유형 | 권장 모델 | 이유 |
|----------|----------|------|
| 파일 탐색, 코드 검색, grep | `haiku` | 반복적·기계적 작업, 추론 불필요 |
| 일반 코딩, 문서 작성, 데이터 변환 | `sonnet` | 속도·비용·품질 균형 |
| 아키텍처 설계, 보안 분석, 복잡한 추론 | `opus` | 높은 추론 품질 필요 |

**팀 구성 최적화 예시 (리서치 팀):**
```
오케스트레이터: sonnet  (팀 조율, 통합)
web-researcher: haiku   (웹 검색, 수집)
data-analyst:   sonnet  (분석, 패턴 발견)
fact-checker:   sonnet  (검증, 교차 확인)
doc-writer:     haiku   (문서 작성, 포맷)
```
opus는 팀원 전체에 쓰지 않고, 아키텍처·보안 판단이 필요한 에이전트에만 적용.

> Claude Code 에이전트 모델 설정: `.claude/agents/{name}.md` frontmatter `model:` 필드
> 상세 기준: `references/agent-design-patterns.md` "모델 선택 가이드"

## 5. 지속적 모니터링 패턴

하네스를 일회성으로 구성하고 끝내지 않는다. 실행 후 다음 지표를 주기적으로 확인한다.

### 5-1. 스킬 크기 모니터링

하네스 실행 후 생성된 SKILL.md 파일들의 줄 수를 확인한다:

```bash
# 모든 SKILL.md 줄 수 확인
find . -name "SKILL.md" | xargs wc -l | sort -rn

# 500줄 초과 파일만 표시
find . -name "SKILL.md" -exec awk 'END{if(NR>500)print NR, FILENAME}' {} \;
```

**모니터링 주기:** 스킬 추가·수정 후, 하네스 진화(Phase 7) 실행 시

### 5-2. Handoff 파일 크기 모니터링

```bash
# artifacts/ 또는 _workspace/ 파일 크기 확인
find artifacts/ _workspace/ -name "*.md" 2>/dev/null | xargs wc -l | sort -rn | head -20

# 200줄 초과 파일 표시
find artifacts/ _workspace/ -name "*.md" 2>/dev/null \
  -exec awk 'END{if(NR>200)print NR"줄", FILENAME}' {} \;
```

### 5-3. 토큰 예산 추정

하네스 실행 전 예상 토큰 비용을 추정한다:

```
예상 토큰 = Σ(각 에이전트가 읽는 파일 줄 수 × 에이전트 수 × 컨텍스트 배율)

컨텍스트 배율:
- context: empty  → 1.0 (작업 관련 파일만)
- context: fork   → 2.0~5.0 (이전 대화 포함)
```

**예시 계산:**
```
3인 팀, context: empty
- 오케스트레이터 SKILL.md: 300줄
- 에이전트 정의 × 3: 100줄 × 3 = 300줄
- work skills × 3: 200줄 × 3 = 600줄
- handoff input: 50줄
총 예상: ~1,250줄 × 에이전트 수에 따른 배율
```

## 6. 최적화 사이클

하네스 운영 중 다음 사이클로 지속 최적화한다.

```
[실행] → [측정] → [발견] → [최적화] → [검증] → [실행]
```

### 사이클 체크리스트

**측정 (실행 직후):**
- [ ] SKILL.md 줄 수 확인 (500줄 초과 없는지)
- [ ] Handoff 파일 크기 확인 (200줄 초과 있는지)
- [ ] 실제 로드된 references/ 파일 수 확인

**발견 (패턴 분석):**
- [ ] 매번 전체 references/를 로드하는 스킬 → 조건부 로딩으로 개선
- [ ] 모든 에이전트에 opus 사용 → 작업별 모델 분리
- [ ] `context: fork` 남용 → `context: empty` + 파일 handoff로 전환
- [ ] 동일 내용이 여러 파일에 중복 → 단일 source of truth로 통합

**최적화 (변경):**
- [ ] 비대해진 SKILL.md → references/ 분리
- [ ] 비대해진 handoff 파일 → 요약본/상세본 분리
- [ ] 중복 스킬 → 공통 스킬로 추출 (`references/skill-writing-guide.md` §9)

**검증 (변경 후):**
- [ ] 최적화 후 동일 프롬프트로 재실행 → 품질 저하 없는지 확인
- [ ] 예상 토큰 예산 재계산 → 개선량 수치화

## 7. 자동 토큰 사용량 측정 및 기록

하네스 실행 중 토큰 비용을 자동으로 추적하고 표준 파일로 저장한다.
오케스트레이터가 각 Phase 완료 시점에 기록을 누적하여, 실행 후 최적화 근거로 활용한다.

### 7-1. 저장 위치

| 플랫폼 | 경로 |
|--------|------|
| Claude Code | `_workspace/token-audit.md` |
| Cursor / Gemini / Codex | `artifacts/token-audit.md` |

아카이브 대상에 포함한다 — 이전 실행의 토큰 이력은 `_workspace_archive/{TS}/token-audit.md` 또는 `artifacts/archive/{TS}/token-audit.md`에 보관.

### 7-2. 기록 시점

| 시점 | 기록 내용 |
|------|----------|
| **Phase 0 완료** | 실행 전 토큰 예산 추정 (에이전트 목록·모델·입력 파일 크기 기반) |
| **각 Phase 완료** | 해당 Phase에서 생성된 handoff 파일 크기 + 누적 합계 갱신 |
| **최종 정리** | 전체 요약 + 최적화 권고 항목 업데이트 |

### 7-3. 표준 파일 형식 (`token-audit.md`)

```markdown
# Token Audit — {도메인명}
> Run: {YYYY-MM-DD HH:MM} | Platform: {Claude/Cursor/Gemini/Codex}

## 요약
| 지표 | 값 | 상태 |
|------|-----|------|
| 총 입력 파일 규모 | ~{N}줄 | {정상/주의/경고} |
| 에이전트 수 | {N}명 | - |
| 사용 모델 종류 | {opus/sonnet/haiku 조합} | - |
| 최대 단일 에이전트 규모 | {name} (~{N}줄) | {정상/주의/경고} |
| handoff 파일 총 규모 | ~{N}줄 | {정상/주의/경고} |

## 에이전트별 입력 규모
| 에이전트 | 모델 | SKILL.md | agent.md | handoff 입력 | context | 소계(줄) |
|---------|------|----------|----------|-------------|---------|---------|
| {agent-1} | {model} | {N}줄 | {N}줄 | {N}줄 | {fork/empty} | {N} |
| {agent-2} | {model} | {N}줄 | {N}줄 | {N}줄 | {fork/empty} | {N} |
| 합계 | - | - | - | - | - | {N} |

## SKILL.md 크기 감사
| 파일 | 줄수 | 상태 | 조치 |
|------|-----|------|------|
| {skill-name}/SKILL.md | {N}줄 | ✅ 정상 / ⚠️ 주의 / 🔴 위반 | {없음 / references/ 분리 권고} |

## Phase별 handoff 파일 규모
| Phase | 파일 | 줄수 | 상태 |
|-------|------|-----|------|
| Phase 1 | `00-input.md` | {N}줄 | ✅ / ⚠️ / 🔴 |
| Phase 2 | `02_{agent}.md` × {N} | {N}줄 | ✅ / ⚠️ / 🔴 |
| Phase N | `final-report.md` | {N}줄 | ✅ / ⚠️ / 🔴 |

## 최적화 권고
| 우선순위 | 대상 | 발견 사항 | 권고 조치 |
|---------|------|---------|---------|
| 🔴 높음 | {파일명} | {N}줄 — 500줄 초과 | references/ 분리 |
| ⚠️ 중간 | {에이전트} | context: fork 사용 | context: empty + 파일 handoff 검토 |
| ℹ️ 낮음 | {에이전트} | opus 사용 (단순 작업) | sonnet/haiku 전환 검토 |
```

### 7-4. 상태 기준값

| 지표 | ✅ 정상 | ⚠️ 주의 | 🔴 위반/경고 |
|------|--------|--------|------------|
| SKILL.md 줄수 | ~300줄 이하 | 300~450줄 | 450줄 초과 |
| agent.md 줄수 | ~100줄 이하 | 100~150줄 | 150줄 초과 |
| handoff 파일 줄수 | ~150줄 이하 | 150~200줄 | 200줄 초과 |
| 에이전트 소계 (줄) | ~500줄 이하 | 500~800줄 | 800줄 초과 |

### 7-5. 오케스트레이터 통합 지침

오케스트레이터 SKILL.md 작성 시 다음 Phase를 추가한다:

**Phase 0 말미 — 토큰 예산 추정:**
```markdown
#### 토큰 예산 추정
각 에이전트의 입력 파일(SKILL.md + agent.md + handoff 입력) 줄수를 합산하여
`{_workspace 또는 artifacts}/token-audit.md`에 "요약" 및 "에이전트별 입력 규모" 섹션을 초기 작성한다.
추정 소계가 800줄을 초과하는 에이전트가 있으면 Phase 1 진입 전 최적화 여부를 사용자에게 확인한다.
```

**각 Phase 완료 후 — handoff 기록:**
```markdown
Phase 완료 후 생성된 handoff 파일 줄수를 `token-audit.md`의 "Phase별 handoff 파일 규모" 섹션에 추가한다.
200줄 초과 파일 발견 시 즉시 상태를 🔴로 표시하고, 다음 Phase 진입 전 요약본 분리를 검토한다.
```

**최종 정리 Phase — 최적화 권고 생성:**
```markdown
전체 실행이 완료되면 `token-audit.md`의 "최적화 권고" 섹션을 작성한다.
- 기준값을 초과한 항목을 우선순위별로 정리
- 다음 실행 전 개선할 항목을 명시
- token-audit.md를 사용자에게 결과 보고 시 함께 제공
```

### 7-6. 이력 활용

`token-audit.md`는 `harness-evolution.md` §3 변경 이력과 연계한다:
- 최적화 조치를 실행했을 때 변경 이력에 "토큰 최적화 — {내용}" 기록
- 아카이브된 이전 `token-audit.md`와 현재 값을 비교하여 개선량 추적
